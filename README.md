# FastFive

> FASTFIVE 출입증 앱을 사용하다가 휴대폰을 기울이면 공이 자연스럽게 움직이는 화면이 인상적이었습니다. "이걸 Compose로도 만들 수 있을까?"라는 단순한 호기심으로 시작했지만, 구현 과정에서 예상치 못한 버벅거림을 마주했고 원인을 측정하고 개선하는 과정까지 이어진 개인 프로젝트입니다.

---

## 프로젝트 개요

FASTFIVE 출입증 앱을 사용하던 중, 휴대폰을 기울이면 화면 속 공들이 자연스럽게 움직이는 UI가 인상적이었습니다. "이걸 Jetpack Compose로도 구현할 수 있을까?"라는 단순한 호기심으로 동일한 사용자 경험을 직접 만들어보기 시작했습니다.

처음 목표는 단순한 UI 재현이었습니다. 그런데 막상 만들어보니 공이 끊기듯 움직이는 현상이 생겼고, "왜 버벅거릴까?"라는 궁금증이 단순한 UI 재현 프로젝트를 성능 분석 프로젝트로 바꾸는 계기가 되었습니다.

---

## 호기심이 문제 해결로 이어진 과정

1. FASTFIVE 앱의 움직임이 재밌어 보여 직접 만들어보기 시작했습니다.
2. 생각보다 공이 끊기듯 움직이는 문제를 발견했습니다.
3. 원인을 찾기 위해 프레임 드랍을 측정하고 구조를 분석했습니다.
4. `withFrameNanos` 기반으로 갱신 구조를 변경했습니다.
5. 측정 결과를 통해 실제로 성능이 개선된 것을 확인했습니다.

---

## 문제 발견

처음에는 센서 이벤트가 들어올 때마다 Compose 상태를 직접 갱신했습니다. 하지만 공이 끊기듯 움직이는 현상이 발생했고, 원인을 추적해보니 센서 이벤트 타이밍과 화면이 그려지는 시점(vsync)이 서로 어긋나고 있다는 것을 발견했습니다.

실제로 같은 에뮬레이터 환경에서 `adb shell dumpsys gfxinfo`로 측정했을 때, Janky Frame 비율이 **82.6%**, 99th percentile 프레임 타임이 **900ms**에 달해 사실상 "뚝뚝 끊기는" 수준이었습니다.

---

## 어떻게 해결했는지

후보로 두 가지를 검토했습니다.

- **`delay(16L)`로 고정 주기 코루틴 루프를 돌리는 방식**: 구현은 단순하지만 `delay`는 실제 디스플레이의 vsync 신호와 무관하게 동작하기 때문에, 타이밍이 살짝씩 어긋나는 문제를 근본적으로 해결하지 못함.
- **Compose의 `withFrameNanos`로 vsync에 동기화하는 방식**: Compose가 이미 `MonotonicFrameClock`을 통해 vsync 신호를 받고 있으므로, 그 신호에 물리 연산을 얹으면 화면 출력과 정확히 같은 박자로 갱신할 수 있음.

타이밍 정확도가 핵심 문제였기 때문에 `withFrameNanos` 방식을 선택했고, 다음과 같이 구조를 바꿨습니다.

1. 물리 연산을 센서 콜백과 분리해 별도의 로직으로 관리했습니다.
2. `withFrameNanos` 기반 게임 루프를 적용해 화면 갱신 타이밍(vsync)에 맞춰 물리 연산이 실행되도록 변경했습니다.
3. 센서 입력과 화면 렌더링의 책임을 분리해 더 안정적으로 상태를 관리할 수 있도록 구조를 개선했습니다.
4. 충돌 연산을 세분화해 빠르게 움직이는 공이 벽을 통과하는 현상을 방지했습니다.

## 측정 결과

| 항목 | 리팩터링 전 | 리팩터링 후 |
|---|---|---|
| Janky Frame 비율 | 82.6% | **2.97%** |
| 99th percentile 프레임 타임 | 900ms | **32ms** |

같은 에뮬레이터 환경에서 측정한 결과, 체감으로만 느끼던 버벅거림이 실제 수치로도 개선된 것을 확인할 수 있었습니다. 단순히 "잘 되는 것 같다"가 아니라, 측정을 통해 개선 여부를 검증해본 경험이었다는 점이 가장 의미 있었습니다.

## 기술 스택

| 분류 | 기술 | 사용 목적 |
|---|---|---|
| Language | Kotlin | Android 앱 개발 언어 |
| UI | Jetpack Compose | 선언적 UI 및 Canvas 기반 커스텀 드로잉 |
| Timing | `withFrameNanos` | vsync 기반 물리 연산 게임 루프 |
| Sensor | `SensorManager` (TYPE_ACCELEROMETER) | 기기 기울기 입력 수집 |
| Architecture | `AndroidViewModel` | 센서 리스너 생명주기 관리 + 물리 상태 보관 |
| Performance | `adb shell dumpsys gfxinfo` | 프레임 드랍(jank) 정량 측정 및 리팩터링 전후 비교 |

## 회고

처음에는 단순히 화면을 따라 만들어보는 것으로 끝날 줄 알았는데, 예상하지 못한 문제를 마주하고 직접 측정하고 개선해보면서 오히려 더 많은 것을 배울 수 있었습니다. 이후에는 문제가 생기면 감으로 수정하기보다, 먼저 재현하고 원인을 확인해보려는 습관이 생겼습니다.

성능 측정은 에뮬레이터 한 환경에서만 진행했기 때문에, 실제 기기에서의 jank 비율은 다를 수 있어 추가 검증이 필요합니다.
